## 摘要

本文介绍 AI 接触层，这是 AI 原生架构五篇系列论文中的第二篇。虽然论文 1 关注执行运行时的机制，本文则探讨概念性问题：AI 智能与外部世界之间的接口本质是什么？

接触层在智能行为与外部能力之间进行协调，使代理能够通过结构化的能力接口访问模型、工具、服务、数据源和设备。我们提出了区分智能与行动的理论基础，展示了接触层如何连接认知与执行，并将其定位在更广泛的 AI 原生架构中。

我们认为，智能与行动的分离不仅仅是架构上的便利，而是 AI 原生系统的基本设计原则。这项工作提供了一个概念框架，将执行基础设施与 AI 系统如何与现实世界交互这一更深层的哲学问题统一起来。

## 1. 引言

### 1.1 纯智能的局限性

人工智能系统在推理、语言理解和多模态感知方面取得了显著进展。来自 OpenAI、Anthropic、Google 和 DeepSeek 等提供商的大规模模型展现出越来越复杂的认知能力。然而，一个关键限制仍然存在：

> *仅靠智能无法产生现实世界的影响。*

为了让 AI 系统执行有意义的任务，它必须与外部系统交互。例子包括：

- 从知识库中检索信息
- 调用 API 执行操作
- 控制物理世界中的设备
- 执行自动化工作流
- 与数字服务交互

换句话说，**AI 必须将其内部推理与外部行动连接起来**。这种连接就是我们定义的 AI 接触层。

### 1.2 缺失的概念层

虽然执行层和运行时系统解决了实现问题，但存在一个更深层次的概念性问题：AI 智能与外部世界之间的接口本质是什么？

这个问题既有哲学维度，也有实践维度：

- **哲学：** 我们如何理解 AI 系统中思想与行动之间的边界？
- **实践：** 我们如何构建能够可靠连接推理与执行的系统？

接触层概念解决了这两个维度，提供了：

1. 理解 AI 与世界交互的概念框架
2. 实现这种交互的架构模型
3. 连接执行基础设施与更广泛系统设计的统一视角

### 1.3 与执行层论文的关系

本文建立在我们配套工作 [1] 中描述的 AI 执行层架构之上。这些工作之间的关系可以通过 AI 行动层的视角来理解：

**行动层分解：** 行动层包含两个子组件：

- **接触层：** 路由、策略评估、提供者选择（“控制平面”）

**互补重点：** 执行层论文描述了*运行时组件*——能力如何被调用、转换和规范化。本文描述了*接触组件*——路由决策如何制定、策略如何评估、提供者如何选择。

它们共同构成了完整的行动层：一个处理执行的机制，另一个处理路由的逻辑。

## 2. 智能 vs. 行动：一个基本区分

### 2.1 两个基本方面

要理解接触层的作用，我们首先必须区分 AI 系统的两个基本方面：

**智能（认知）：** 内部认知过程包括：

- 对目标和约束的推理
- 行动序列的规划
- 语言和上下文的理解
- 决策制定
- 记忆与学习

这些过程完全发生在 AI 模型或代理内部。智能代表了系统*思考*的能力。

**行动：** 与环境的交互包括：

- 调用 API
- 从数据库中检索数据
- 发送消息
- 控制设备
- 触发自动化

这些过程涉及与外部系统的交互。行动代表了系统*执行*的能力。

### 2.2 为何分离重要

没有行动，智能仍然是孤立的。AI 系统可能对任务进行完美推理，但缺乏执行能力。考虑：

- 一个生成完美计划但无法调用工具的规划系统
- 一个识别最优解决方案但无法实现的推理引擎
- 一个正确做出决策但无法根据决策行动的决策者

智能与行动的分离不仅仅是架构上的——它是根本性的。AI 系统需要在认知与行动之间有一个结构化的接口。

### 2.3 历史背景

这种区分与计算历史上的其他基本分离类似：

- **CPU vs. I/O：** 处理器计算；I/O 设备与外部世界交互
- **算法 vs. 数据结构：** 算法处理；数据结构组织信息
- **控制平面 vs. 数据平面：** 控制做出决策；数据平面转发数据包

类似地：

```
AI 认知层（思考） → AI 接触层（桥梁） → 外部世界（行动）
```

接触层作为内部推理与外部行动之间的结构化边界。

## 3. 接触层概念

### 3.1 定义

**接触层是 AI 智能与外部世界之间的接口。**

概念上：

```
智能行为
    ↓
  接触层
    ↓
  外部世界
```

外部世界包括：

- AI 能力（模型、嵌入、生成）
- 数字服务（API、数据库、自动化平台）
- 知识系统（搜索、检索、知识库）
- 物理设备（物联网、机器人、传感器、执行器）

接触层将内部决策转换为可执行的操作。

**形式化定义：** 接触层被定义为一个元组：

```
CL = (C, Policy, Context)

其中：
- C：可用能力集合 C = {c₁, c₂, ..., cₙ}
- Policy：函数 Policy(c, ctx) → (provider, constraints)
- Context：执行上下文，包括偏好、约束、历史

接触层将代理意图映射到能力执行：

Route: Intent × Context → Capability × Provider × Constraints
```

### 3.2 核心功能

接触层执行几个基本功能：

**1. 能力协调：** 将代理意图转换为能力调用。当代理决定“搜索信息”时，接触层确定调用哪个能力以及使用什么参数。

**2. 提供者抽象：** 屏蔽代理免受提供者特定细节的影响。代理无需知道搜索是由提供者 A 还是提供者 B 执行的——它们表达能力需求，接触层处理提供者选择。

**2.1. 策略评估性能：** 如果为每个请求评估复杂的路由规则，策略评估在高并发下可能成为瓶颈。架构通过以下方式解决这一问题：

- **同步 vs. 异步决策：** 简单的路由决策（基于静态策略或缓存偏好）以最小延迟同步做出。复杂的决策（需要实时成本优化或策略协商）可以异步处理，回退到缓存决策。

- **策略缓存：** 频繁评估的策略决策被缓存。例如，常见能力-提供者组合的路由偏好可以缓存并定期刷新，而不是每个请求重新评估。

- **决策预计算：** 对于可预测的工作负载，可以根据历史模式预计算路由决策，并通过后台进程更新。

- **性能预算：** 接触层为同步路由强制执行最大决策延迟（例如 10 毫秒）。如果策略评估超过此预算，则使用缓存或默认路由。

**3. 行动序列化：** 将多个能力协调成连贯的行动序列。复杂任务需要按特定顺序执行的多个能力，接触层对此进行编排。

**4. 故障策略评估：** 确定对能力故障的适当响应。当能力失败时，接触层评估组织策略，决定是重试、选择替代提供者还是向代理报告失败。实际的重试执行和错误处理机制由执行运行时 [1] 执行。

**5. 上下文管理：** 跨能力调用协调状态持久化。多步任务需要上下文持久化，接触层通过协调规则进行编排。状态存储和检索机制由外部会话存储或执行运行时处理，保持接触层的无状态设计。

**5.1. 状态架构：** 接触层协调状态管理，但本身不存储状态，保持水平可扩展性。架构支持多种状态策略：

- **会话状态存储：** 对于多轮对话，状态通常存储在专用会话存储（如 Redis、内存缓存）中，通过会话 ID 引用。接触层从此存储中读取和写入。

- **无状态操作：** 对于单次调用场景，不需要状态持久化。每次能力调用都是独立的。

- **代理管理状态：** 在某些架构中，代理维护自己的状态，并在每个请求中传递相关上下文。接触层仅传输此上下文。

**水平可扩展性：** 为了保持水平可扩展性，接触层本身保持无状态——它不维护内存中的会话状态。而是：

- 会话标识符随每个请求传递
- 状态持久化到外部存储
- 任何接触层实例都可以处理同一会话的任何请求

这种状态协调（接触层）与状态存储（外部存储）的分离使得弹性扩展成为可能，同时支持多步工作流。

### 3.3 能力作为交互的语言

在接触层内，基本抽象是**能力**。能力代表 AI 系统可以调用的结构化操作。

能力示例：

- `search(query: string) → SearchResult[]`
- `send_email(to: string, subject: string, body: string) → Status`
- `control_device(device_id: string, command: string) → DeviceResponse`
- `call_api(endpoint: string, payload: object) → ApiResponse`

能力是结构化的、类型化的、可发现的。它们提供：

1. **接口抽象：** 统一表示不同的操作
2. **类型安全：** 强类型输入和输出
3. **可发现性：** 代理可以查询可用能力
4. **可组合性：** 能力可以组合成工作流

能力作为 AI 系统与外部世界交互的语言，而接触层作为这种语言的翻译器和协调器。

## 4. 架构实现

### 4.1 从概念到架构

接触层概念映射到具体的架构组件。实现包括：

**核心组件：**

1. **路由引擎：** 将意图和上下文映射到能力和提供者
2. **策略评估器：** 应用组织策略（成本、性能、安全）
3. **提供者注册表：** 维护可用能力和提供者的目录
4. **会话协调器：** 管理跨能力调用的上下文

**接口边界：**

接触层暴露两个主要接口：

1. **北向接口：** 与代理智能层通信，接收意图和上下文
2. **南向接口：** 与执行运行时通信，发送能力请求

这种分离保持了关注点分离：接触层专注于路由决策，执行运行时专注于执行机制。

### 4.2 澄清术语边界

澄清术语对于避免混淆至关重要：

**执行运行时 vs. 执行层：**

- **执行运行时：** 执行能力调用的组件（来自论文 1）
- **执行层：** 更广泛的类别，包括运行时和协调基础设施

**接触层 vs. 编排层：**

- **接触层：** 连接智能与行动的接口（本文重点）
- **编排层：** 可能包括接触层和额外的协调逻辑

**行动层分解：**

行动层可以分解为：

```
Action Layer = Contact Layer + Execution Runtime
```

这种分解澄清了论文 1 和论文 2 之间的关系：论文 1 描述执行运行时，论文 2 描述接触层，它们共同构成行动层。

### 4.3 精确的定义

为了精确，我们定义：

**定义 1（接触层）：** 接触层是 AI 智能与外部世界之间的接口，它将代理意图映射到能力执行，应用策略进行提供者选择，并协调跨能力调用的上下文。

**定义 2（能力）：** 能力是一个结构化的操作，具有明确定义的输入和输出模式，可以由 AI 系统执行。

**定义 3（提供者）：** 提供者是能力的实现者，可以是模型、API、服务或设备。

接触层因此定义为一个映射函数：

```
contact: Intent × Context × Policies → Capability × Provider × Parameters
```

这种形式化强调了接触层的转换角色：它将内部认知状态（意图）转换为外部可执行操作（能力调用）。

## 5. 数字与物理交互

### 5.1 双域支持

接触层的一个关键见解是它统一支持数字和物理交互：

**数字域：**
- 模型推理（文本生成、图像分析）
- API 调用（REST、gRPC、GraphQL）
- 数据操作（查询、转换、存储）
- 消息传递（电子邮件、通知、聊天）

**物理域：**
- 设备控制（IoT、机器人、传感器）
- 物理动作（移动、操纵、导航）
- 环境交互（温度、照明、安全）
- 人机交互（语音、手势、触摸）

这种双域支持对于构建全面交互的 AI 系统至关重要。AI 系统不仅应能处理数字信息，还应能在物理世界中行动。

### 5.2 物理交互的意义

物理交互引入了独特的挑战：

1. **延迟敏感性：** 物理控制通常有严格的实时要求
2. **安全性：** 物理行动可能产生不可逆的后果
3. **可靠性：** 物理系统比数字服务更容易出现故障
4. **上下文感知：** 物理环境复杂且动态变化

接触层通过以下方式解决这些挑战：

- **实时路由：** 为物理能力选择具有足够低延迟的提供者
- **安全策略：** 对物理行动实施更严格的安全检查
- **冗余设计：** 为关键物理能力支持多个提供者
- **环境上下文：** 在路由决策中包含物理环境因素

### 5.3 边缘计算的影响

接触层架构自然支持边缘计算场景：

**边缘接触点：** 接触层实例可以部署在边缘，靠近物理设备，以最小化延迟。

**分层路由：** 决策可以在云和边缘之间分层做出，平衡集中策略与本地优化。

**离线操作：** 边缘接触点可以在网络中断时继续操作，使用本地提供者和缓存策略。

这种灵活性使得接触层适用于从云中心到完全边缘部署的各种场景。

## 6. 行动接口定义

### 6.1 形式化定义

行动接口是接触层与执行运行时之间的边界。我们形式化定义行动接口：

**定义 4（行动接口）：** 行动接口是接触层与执行运行时之间的标准化通信协议，用于请求能力执行和接收结果。

行动接口支持以下操作：

1. **invoke(capability: string, input: object, context: Context): Promise<Result>**
   - 请求能力执行
   - 返回包含输出或错误的结果

2. **discover(): CapabilityDescription[]**
   - 查询可用能力
   - 返回能力描述列表

3. **status(request_id: string): ExecutionStatus**
   - 检查异步执行的状态
   - 返回进度、结果或错误信息

### 6.2 精确性声明

行动接口旨在：

**精确性：** 明确定义的操作和错误条件
**可扩展性：** 支持新能力和提供者
**互操作性：** 跨不同运行时实现工作
**可观测性：** 提供执行遥测数据

这种精确性使得接触层能够与各种执行运行时集成，同时保持一致的接口。

### 6.3 架构意义

行动接口的架构意义：

**标准化边界：** 定义接触层和执行运行时之间的清晰边界
**解耦进化：** 允许各层独立进化
**测试隔离：** 支持各层独立测试
**实现多样性：** 支持不同的运行时实现（Rust、Python、TypeScript 等）

通过标准化这一接口，我们实现了论文 1 和论文 2 所描述组件之间的清晰分离。

## 7. 与现有框架的比较

### 7.1 当前方法

当前 AI 系统采用多种方法连接智能与行动：

**直接集成：** 代理直接调用特定提供者 API
- 优点：简单直接
- 缺点：紧密耦合、缺乏灵活性

**编排框架：** 使用 LangChain、LlamaIndex 等框架
- 优点：抽象复杂性、工具丰富
- 缺点：框架锁定、运行时混合

**自定义中间件：** 构建专门的中介层
- 优点：定制化、控制力强
- 缺点：开发成本高、维护负担

### 7.2 区别性方法

接触层方法与当前方法的区别：

1. **关注点分离：** 明确分离路由（接触层）与执行（运行时）
2. **最小性：** 接触层专注于路由决策，不包含执行逻辑
3. **协议驱动：** 基于标准化协议而非框架特定 API
4. **水平可扩展：** 无状态设计支持弹性扩展

### 7.3 集成路径

接触层可以集成到现有系统中：

**渐进采用：** 从直接集成开始，逐步引入接触层进行路由决策
**并行运行：** 接触层与现有编排框架并行运行，逐步迁移
**完全替换：** 用接触层+执行运行时替换现有框架

集成路径允许组织根据其具体需求和约束采用接触层架构。

### 7.4 与多智能体框架的比较

| 框架 | 重点 | 与接触层的关系 |
|------|------|---------------|
| LangChain | 工具调用、工作流编排 | 接触层可以替换 LangChain 的调用逻辑，提供更解耦的架构 |
| AutoGPT | 自主目标追求 | 接触层提供结构化接口，AutoGPT 代理通过此接口与外部能力交互 |
| CrewAI | 多智能体协作 | 接触层协调智能体与外部能力之间的交互，保持智能体专注于协作 |
| Microsoft Autogen | 多智能体对话 | 接触层作为智能体与工具/API 之间的中介层 |

接触层补充了这些框架，提供了标准化的能力接口和路由层，而不强制特定的智能体架构。

## 8. AI 系统的演进

### 8.1 历史进程

接触层的引入反映了 AI 系统设计中更广泛的演进：

**阶段 1：模型智能（2017-2022）**

重点：改进模型能力  
示例：GPT-2、GPT-3、BERT、T5  
特征：模型作为独立能力

**阶段 2：能力集成（2022-2025）**

重点：将模型连接到工具和服务  
示例：ChatGPT 插件、LangChain、AutoGPT  
特征：模型具备工具使用能力

**阶段 3：环境交互**

重点：AI 系统与环境交互  
示例：AI 原生应用、自主系统、边缘 AI  
特征：行动层、能力生态系统  
状态：当前和新兴阶段

这一演进可以总结为：

```
智能 → 能力 → 接触层 → 行动
```

**长期（5+ 年）：**

接触层代表了 AI 系统设计的关键演进步骤，从孤立的智能到与环境深度集成的系统。

## 9. 个人 AI 基础设施

### 9.1 概念

个人 AI 基础设施使个体能够拥有和控制其 AI 系统。接触层在这一愿景中扮演关键角色：

**个人接触层：** 个人拥有的接触层实例，根据个人偏好和策略路由能力请求。

**能力主权：** 个人选择哪些能力可用，哪些提供者被信任，以及应用什么策略。

**隐私保护：** 接触层可以在本地运行，确保敏感数据不离开个人设备。

### 9.2 架构

个人 AI 基础设施架构：

```
个人设备
├── 个人代理（推理、规划）
├── 个人接触层（路由、策略）
├── 本地能力（设备 API、本地模型）
└── 远程能力（云服务、外部 API）
```

接触层协调本地和远程能力，根据个人策略和隐私偏好做出路由决策。

### 9.3 部署场景

**完全本地：** 所有组件在个人设备上运行
**混合：** 接触层在本地，某些能力在云中
**去中心化：** 接触层和能力分布在多个设备上

接触层的灵活性支持这些不同场景，适应不同的隐私、成本和性能要求。

### 9.4 个人 AI 用例：智能家居 + 电子邮件

考虑一个个人 AI 代理管理智能家居和电子邮件的场景：

```
代理意图："安排明天上午 9 点的会议，并调低恒温器"

接触层路由：
1. 将"安排会议"路由到日历能力
2. 将"调低恒温器"路由到智能家居能力

执行：
1. 日历能力调用 Google Calendar API
2. 智能家居能力调用 Nest Thermostat API

结果：
- 会议安排在日历中
- 恒温器温度降低
```

接触层确保正确的能力被调用，适当的提供者被选择，并且操作根据个人偏好执行。

接触层在 AI 原生栈中的位置：

                 │

                 │

### 10.2 概念统一

**统一策略：** 跨所有能力应用一致的策略
**统一可观测性：** 为所有交互提供一致的遥测数据

这种统一简化了 AI 系统设计，减少了跨不同交互模式的认知负担。

## 11. 哲学意义

### 11.1 AI 中的思想与行动

接触层概念引发了关于 AI 系统中思想与行动的哲学问题：

**认知边界：** 智能在哪里结束，行动从哪里开始？
**意向性：** AI 系统能否拥有真正的意向，还是仅仅模拟意向？
**责任：** 当 AI 系统通过接触层行动时，谁对结果负责？

这些问题不仅仅是学术性的——它们直接影响我们如何构建与世界交互的 AI 系统。

### 11.2 协调者角色

接触层作为协调者，调解：

**内部与外部：** 将内部认知状态转换为外部行动
**抽象与具体：** 将高级意图转换为具体操作
**策略与机制：** 将策略决策与执行机制分离

这种协调者角色对于构建既智能又可行动的 AI 系统至关重要。

### 11.3 对 AI 安全的意义

接触层通过以下方式增强 AI 安全：

**行动拦截：** 在行动执行前拦截和检查
**策略执行：** 强制执行安全策略
**审计跟踪：** 记录所有交互以供审计

通过控制 AI 系统与世界的接触点，接触层提供了安全关键操作的关键控制点。

### 11.4 行动拦截与安全约束

接触层可以实现行动拦截机制：

**预执行检查：** 在执行前验证行动是否符合策略
**运行时监控：** 在行动执行时监控合规性
**后执行审计：** 记录行动以供事后分析

这些机制在涉及不可逆行动的场景中特别重要。

### 11.5 信任域与批量授权

接触层支持信任域概念：

**域边界：** 定义不同的信任级别（完全信任、部分信任、不信任）
**授权策略：** 根据信任域应用不同的授权策略
**域转换：** 在域之间安全地转换上下文

批量授权允许在单个策略决策下授权多个相关行动，减少延迟同时保持安全。

### 11.6 物理行动预执行模拟

对于物理行动，接触层可以支持：

**数字孪生模拟：** 在物理执行前在数字孪生中模拟行动
**后果分析：** 分析行动的潜在后果
**安全验证：** 验证行动在安全约束内

这种模拟能力对于安全关键的物理交互特别有价值。

## 12. 结论

接触层代表了 AI 系统设计的基本演进。通过明确分离智能与行动，它提供了：

1. **概念清晰性：** 澄清了 AI 智能与外部世界之间的边界
2. **架构解耦：** 分离路由决策与执行机制
3. **实践可行性：** 支持从个人 AI 到企业系统的各种应用

接触层不仅是一个架构组件，更是一个概念框架，指导我们思考 AI 系统如何与世界交互。它将执行基础设施（论文 1）与更深层的哲学问题联系起来：AI 系统如何有意义地行动？

随着 AI 系统变得更加复杂和强大，接触层等结构化接口将变得更加重要。它们提供了必要的协调，使 AI 系统能够智能地行动，而不仅仅是智能地思考。

interface RoutingRequest {
  // Agent intent

  context: {

  };
}
