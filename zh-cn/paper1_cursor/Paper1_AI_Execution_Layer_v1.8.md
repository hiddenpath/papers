**作者：** 王鲁强

**预印本版本：** 1.8

## 摘要

AI 能力的快速增长导致了碎片化的集成环境，应用程序必须通过不兼容的 API 连接异构的提供者、工具、服务和设备。本文提出了 AI 执行层，一种最小运行时架构，用于标准化能力执行过程，同时将编排逻辑隔离在运行时边界之外。系统形式化为三层逻辑模型 `S = (A, P, E)`，其中 `A` 表示认知层，`P` 表示接触层（路由与策略），`E` 表示执行层。该逻辑模型映射到更广义的五层 AI 原生实现栈，从而分离架构不变量与部署结构。能力统一形式化为 `C = (I, O, Σ, Q, P)`，其中 `I` 和 `O` 为模式，`Σ` 定义执行语义，`Q` 定义质量遥测数据，`P` 定义定价元数据。执行运行时收集并传输 `Q` 与定价元数据，但不参与策略决策。最小性作为设计约束被强制实施：运行时仅执行确定性功能，包括协议转换、能力调用、结果标准化、连接管理、针对瞬时故障的有界重试语义、错误传播和会话上下文转发。通过排除规划、工作流编排和提供者选择，运行时保持轻量、无状态和水平可扩展，同时支持流式传输、基于轮询的异步任务、输入验证和基础安全控制。

**关键词：** AI 原生架构、执行层、能力抽象、运行时系统、AI 基础设施

## 1. 引言

### 1.1 集成挑战

AI应用日益依赖模型供应商、外部服务和本地设备所提供的异构能力。OpenAI [1]、Anthropic [2]、Google及DeepSeek等提供商通过API公开强大的模型，但直接集成会导致紧密耦合，并使系统复杂性不断增加。

当前多数AI框架通过应用级编排（如LangChain [3]）应对这一挑战。虽然这种方法提升了开发效率，但它常将编排策略与执行机制耦合于同一运行时中，导致框架锁定并降低了组件的可重用性。

### 1.2 缺失的层

当前生态系统仍缺乏一个专用于能力调用的最小执行层。该层应具备以下功能：

1. 执行由标准化协议定义的能力
2. 将协议请求转化为特定提供者的API调用
3. 将结果标准化为统一的响应格式
4. 处理执行错误，但不实现应用级的重试策略

该层排除了规划、工作流编排和应用逻辑等高级关注点，从而保持执行边界的轻量化和可重用性。

### 1.3 贡献

本文的主要贡献如下：

1. **形式化的架构模型：** 提出了三层逻辑模型 `S = (A, P, E)`，以分离认知、编排和执行，并明确了各层的职责。
2. **最小运行时约束：** 将最小性形式化为一个可验证的设计约束，明确了运行时的职责与非职责，以支持确定性的能力执行。
3. **统一的能力语义：** 将能力形式化为 `C = (I, O, Σ, Q, P)`，使得技术调用和经济可观测性能够共享单一的模式级对象。
4. **跨层集成：** 将三层逻辑模型映射到五层AI原生实现栈，阐明了执行层不变量在完整系统部署中如何保持。

执行层随后将被定位在后续章节中发展的更广泛的五层AI原生架构中。

```
+------------------------------------------------------+
| Agent Intelligence Layer                             |
| (Reasoning, Planning, Cognition)                     |
+------------------------------------------------------+
                        ↓
+------------------------------------------------------+
| Contact Layer                                        |
| (Routing, Provider Selection, Workflow Control)      |
+------------------------------------------------------+
                        ↓
+------------------------------------------------------+
| AI Execution Layer (Runtime)                         |
| - Protocol → API Translation                         |
| - Result Normalization                               |
| - Connection Management                              |
| - Error Propagation                                  |
| - Context Forwarding                                 |
+------------------------------------------------------+
                        ↓
--- AI Protocol Boundary ---
                        ↓
+------------------------------------------------------+
| Capability Providers / Tools / Devices               |
| (LLMs, APIs, Databases, External Services, IoT)      |
+------------------------------------------------------+
```

## 2. 形式化架构模型与问题阐述

### 2.1 逻辑模型：`S = (A, P, E)`

AI执行架构被定义为以下逻辑元组：

```
S = (A, P, E)
```

其中：

- `A` 表示**代理智能层**（推理和规划），
- `P` 表示**接触层**（路由和决策），
- `E` 表示**执行层**（能力调用）。

该模型将AI系统分离为三个不同层次，每层都有明确定义的职责：

**代理层（A）：** 代理体现智能行为。它们对任务进行推理、规划行动序列，并决定应调用哪些能力。该层处理：

- 理解用户目标
- 规划行动序列
- 对能力需求做出决策
- 管理对话上下文与记忆

**接触层（P）：** 接触层决定能力*如何*被选择与组合。该层负责：

- 基于代理意图的能力选择
- 提供者偏好与选择
- 路由策略
- 成本或延迟优化
- 策略的执行

**执行层（E）：** 执行层*仅*负责能力调用的执行。它不实现路由策略、提供者选择或代理逻辑。这种设计使运行时保持最小化且可重用。

### 2.2 逻辑模型与实现栈

在本文中，`S = (A, P, E)` 是**逻辑模型**。后续提出的五层AI原生栈是**实现栈**，用于在生产系统中实现该模型。

- 逻辑模型规定了系统的不变量与边界。
- 实现栈规定了部署接口和工程组合方式。

这种区分避免了一个常见的建模冲突：系统在实现上可以是五层的，而在执行逻辑上仍保持三层结构。

### 2.3 分离的理由

这种分离解决了现有框架中存在的关键架构问题。许多系统将编排职责与执行机制相混淆，导致：

- **框架锁定：** 应用程序与特定编排框架紧密耦合
- **有限的可重用性：** 运行时组件不能轻易跨不同应用程序重用
- **复杂度蔓延：** 执行运行时增长以包含策略逻辑，使其难以维护

通过分离这些关注点，我们实现了以下优势：

- **运行时可移植性：** 执行运行时可在不同编排策略间重用
- **策略灵活性：** 组织可实现自定义路由策略而无需修改运行时
- **清晰边界：** 每个组件都有明确定义的输入、输出及职责

## 3. 为什么最小执行很重要

最小性被视为一种系统约束，而非风格偏好。将策略逻辑嵌入执行运行时会引入状态耦合、降低可移植性并增加跨层故障传播。因此，最小性被定义如下：

> **最小性约束：** 执行层必须仅实现确定性能力调用，且不得执行策略驱动的决策，如提供者选择、策略性重试或工作流级回退。

在此约束下，运行时语义通过核心职责和明确的非职责来界定。

### 3.1 核心职责

运行时作为标准化的能力执行引擎。其职责被有意限制在以下六个核心功能：

**1. 能力调用：** 执行AI协议中定义的能力。运行时接收标准化的能力请求，并调用适当的执行路径。

**2. 协议翻译：** 将协议级请求转换为特定提供者的API调用。该翻译层使应用程序能够使用标准化的能力定义工作，而运行时处理特定提供者的实现细节。

**3. 结果标准化：** 向调用者返回标准化的响应。无论哪个提供者执行了能力，运行时都将结果标准化为一致的格式。

**4. 连接处理：** 管理与提供者的网络通信。包括连接池、超时管理和网络级错误处理。

**5. 错误传播：** 报告执行错误，但不实现应用级的重试策略。运行时区分执行错误（由其报告）和重试逻辑（后者属于编排层）。

**5.1. 有界重试语义（可选）：** 对于瞬态网络故障（如连接超时、429速率限制），运行时可以实现有限的微重试——通常为1-2次立即重试，在网络层采用指数退避机制。这与应用级重试策略的不同之处在于：

- 微重试是有界且自动的
- 它们仅解决瞬态网络级故障
- 它们不实现策略驱动的重试逻辑（如提供者回退）
- 总重试持续时间必须在执行超时预算内

这一选择反映了生产约束：瞬态网络故障很常见，将每个瞬态事件上报至编排层会引入不必要的延迟和跨层通信开销。

**6. 会话传输：** 转发会话上下文，但不解释应用策略。运行时传输能力执行所需的上下文信息，但不解释或修改策略决策。

### 3.2 运行时不做的事

为保持其最小化设计，运行时明确避免以下事项：

- **代理规划：** 决定调用什么能力序列
- **工作流编排：** 管理复杂的多能力工作流
- **应用逻辑：** 实现特定于应用程序的业务逻辑
- **提供者选择：** 选择哪个提供者应执行能力
- **路由策略：** 实现成本优化或基于延迟的路由
- **应用级重试策略：** 确定何时以及如何跨提供者或工作流重试失败的执行

这些职责属于编排层和代理层，以保持关注点的清晰分离。

### 3.3 无状态执行设计

运行时作为无状态执行引擎运行。每次能力调用都是独立的，任何必要的状态由调用的代理或编排层维护。这种设计实现了以下优势：

- **水平可扩展性：** 可部署多个运行时实例而无需状态同步
- **简化调试：** 每次执行都是自包含且可追溯的
- **故障隔离：** 单次执行中的故障不影响其他执行

### 3.4 异步和长时间运行的任务

现代AI能力超越了同步文本生成，包含了视频生成、大规模搜索索引和多步文档处理等长时间运行的任务。执行层在保持最小化设计的同时，适应了以下模式：

**流式响应：** 对于支持服务器发送事件（SSE）或WebSocket连接的能力，运行时：

- 代表调用者建立流式连接
- 转发流数据块而不缓冲整个响应
- 立即报告流错误，无缓冲延迟
- 允许编排层处理流取消操作

**基于轮询的API：** 对于返回作业ID的异步能力，运行时：

- 执行初始API调用并返回作业标识符
- 将轮询逻辑委托给编排层（后者具有策略上下文）
- 为轮询查询提供独立的 `check_status` 操作
- 不实现自动轮询或状态变更通知机制

**设计理由：** 通过将长时间运行的任务状态管理保持在编排层，运行时保持无状态，不需要后台作业跟踪机制。这保留了水平可扩展性并简化了故障恢复。

### 3.5 安全考虑

作为AI认知与外部能力之间的行动接口，执行层在安全模型中占据关键位置。虽然运行时有意避免业务逻辑，但它应实现基本的安全措施：

**输入验证：** 运行时在执行前根据协议模式验证能力请求。所有 `input_schema` 定义必须符合 JSON Schema 2020-12 或更高版本，确保跨实现的互操作性和标准化验证。运行时在调用时执行模式验证，在结构错误到达提供者之前捕获它们。

**PII处理（可选）：** 组织可以在运行时层配置PII剥离或编辑功能。这是针对试图渗出敏感数据的提示注入攻击的纵深防御措施。然而，这是一个可配置选项——主要的PII保护仍是代理层和应用层的责任。

**速率限制：** 运行时可以实现本地速率限制，以防止失控的能力调用，保护运行时基础设施和下游提供者。

**审计日志：** 所有能力调用都应被记录以进行安全审计，包括能力名称、提供者、时间戳和执行状态（成功/失败）。

为使编排层能够进行观测，运行时应在标准化的响应元数据中包含以下指标：

- `retry_count`：执行期间执行的微重试次数（如适用）
- `translation_latency_ms`：协议翻译所花费的时间
- `execution_latency_ms`：实际提供者API调用所花费的时间

这些指标允许接触层评估提供者稳定性并做出明智的路由决策。

**安全边界：** 执行层本身不是安全边界。它在可信环境中运行，其中编排层已经认证代理并授权能力请求。运行时的安全角色是提供纵深防御，而非主要的访问控制。

## 4. 理论基础：四大核心原则

### 4.1 原则一：能力抽象原则

**原则：** *AI代理应依赖能力，而非具体的提供者或API。*

**形式化定义：** 能力 `C` 定义为：

```
C = (I, O, Σ, Q, P)
```

其中：

- `I`：输入模式
- `O`：输出模式
- `Σ`：执行语义
- `Q`：质量遥测模式（如延迟、retry_count、可靠性信号）
- `P`：定价和计费模式（如单价、计费模式、使用计数器）

提供者P实现能力C。代理通过以下方式调用能力：

```
Invoke(C, Input) → Output
```

而非直接调用特定提供者的API。

**意义：** 该原则实现了代理与提供者之间的解耦。代理表达其需要什么（能力）而非如何获得它（特定API）。这种抽象实现了以下优势：

- 无需修改代理即可替换提供者
- 多提供者能力实现
- 标准化能力发现和调用

**与API的区别：** 理解根本区别至关重要：

| 概念 | 本质 | 角色 |
|------|------|------|
| 函数 | 计算 | 算法操作 |
| API | 远程接口 | 访问机制 |
| 工具 | 代理可调用函数 | 调用接口 |
| **能力** | **可执行能力** | **现实世界行动** |

能力代表的是能力本身（可以做什么），而API代表的是访问机制（如何调用）。这种区别对AI原生架构至关重要。

**与MCP的关系：** 模型上下文协议（MCP）定义了模型-工具接口，指定了AI模型如何发现和调用工具。相比之下，能力定义了运行时-提供者接口，指定了执行运行时如何调用提供者实现。这些接口是互补的：执行层可以将MCP作为其能力协议之一实现，将MCP工具定义翻译为特定提供者的实现。在本文范围内，执行运行时不优化 `Q` 或 `P`；它仅测量、标准化并转发这些字段，供编排层进行策略决策。

### 4.2 原则二：认知-行动分离原则

**原则：** *AI认知与能力执行应由行动层分离。*

**架构：**

```
认知层（推理、规划、记忆）
        ↓
行动层（路由、执行）
        ↓
能力提供者
```

**理由：** 若无此分离，代理必须直接调用API、工具和服务，从而导致：

- 与特定提供者的紧耦合
- 跨不同执行环境的低可移植性
- 生态系统碎片化

行动层在认知过程（决定做什么）和执行过程（执行行动）之间进行中介，使代理能够专注于推理，而执行层则处理能力调用的具体机制。

### 4.3 原则三：能力路由原则

**原则：** *能力执行应基于上下文进行动态路由。*

**路由函数：**

```
Route(C, Context) → Provider
```

**上下文参数：**

- **延迟：** 响应时间要求
- **成本：** 预算约束
- **策略：** 组织或监管策略
- **可用性：** 提供者正常运行时间和可靠性
- **安全：** 数据处理和隐私要求

**示例：** 对于能力 `translate`，可用提供者可能包括：

- 提供者 A：较低成本，较高延迟
- 提供者 B：较高成本，较低延迟

路由系统根据上下文动态选择。如果用户优先级是速度，选择提供者 B。如果成本优化最重要，选择提供者 A。

**类比：** 该机制类似于DNS路由或服务网格路由 [7]，但被路由的对象是能力而非网络请求。

### 4.4 原则四：能力图执行

**原则：** *复杂AI任务可以建模为能力图上的执行。*

**形式化定义：** 能力图G定义为：

```
G = (Capabilities, Dependencies)
```

为确保执行的安全性和确定性排序，我们将 `G` 约束为有向无环图（DAG）。不允许存在循环。

**示例：** 任务"撰写并发布博客文章"对应的能力图为：

```
search → summarize → generate_text → translate → publish
   ↓         ↓            ↓             ↓          ↓
[exec]    [exec]       [exec]        [exec]     [exec]
```

**执行过程：**

1. 代理对任务进行推理，并构建能力图
2. 编排层确定执行顺序和提供者选择
3. 执行运行时按顺序调用每个能力
4. 结果流经能力图，每个能力从其前驱接收输入

该模型使复杂任务能够被分解为可组合的能力序列，执行运行时则处理每个能力调用的具体机制。

## 5. AI 协议与能力定义

### 5.1 协议角色

AI协议定义了如何描述能力。它指定了以下内容：

- **提供者：** 提供AI或非AI能力的实体
- **模型：** 具有定义能力的特定AI模型
- **能力类型：** 能力类别（聊天、嵌入、图像生成等）
- **端点：** 能力可被调用的位置
- **参数：** 输入和输出模式

协议作为编排与执行之间的契约边界：编排层决定策略，执行层强制执行调用语义。

协议通常表示为结构化清单：

```
provider/
├── metadata.yaml
└── models/
    ├── model_a.yaml
    └── model_b.yaml
└── capabilities/
    ├── chat.yaml
    └── embedding.yaml
```

### 5.2 能力定义

每个能力被定义为包含以下要素：

- **名称：** 语义标识符（如 `generate_text`、`search_web`）
- **输入模式：** 结构化的输入规范
- **输出模式：** 结构化的输出规范
- **执行语义：** 调用和运行时行为契约
- **质量元数据（`Q`）：** 执行时发出的测量质量字段
- **定价元数据（`P`）：** 执行时发出的计量和计费字段
- **提供者元数据：** 关于提供此能力的提供者信息

**能力定义示例：**

```yaml
name: generate_text
description: Generate text based on a prompt
input_schema:
  type: object
  properties:
    prompt:
      type: string
      description: The input prompt
    max_tokens:
      type: integer
      description: Maximum tokens to generate
  required: [prompt]
output_schema:
  type: object
  properties:
    text:
      type: string
      description: Generated text
execution:
  endpoint: /v1/completions
  method: POST
quality:
  telemetry_fields: [execution_latency_ms, retry_count]
pricing:
  billing_model: per_token
  currency: USD
provider:
  name: Example Provider
  version: 1.0.0
```

### 5.3 以提供者为中心的组织

一个重要的设计决策是能力描述保持以提供者为中心，而非以能力为中心。清单结构遵循：

```
Provider → Models → Capabilities
```

而非：

```
Capability → Providers
```

运行时可以动态构建能力索引：

```
chat/
├── provider A
├── provider B
└── provider C
embedding/
├── provider A
└── provider D
```

该索引可以定期重建，并基于以下因素进行优化：

- 用户偏好
- 延迟测量
- 成本计算
- 可靠性指标

### 6.2 行动层分解

行动层包含两个关键子组件：

**接触层（路由/策略）：**

- 能力路由
- 提供者选择
- 策略评估
- 环境选择

**执行运行时（调用）：**

- API调用
- 工具执行
- 错误处理
- 结果标准化

这种分解保持了关注点的清晰分离：接触层负责*决策*（选择哪个提供者、哪个环境），而执行运行时负责执行*行动*（调用API、返回结果）。

当映射到逻辑模型 `S = (A, P, E)` 时，接触层对应 `P`，执行运行时对应 `E`。

### 6.3 定义声明

执行层定义如下：

> *执行层是AI认知与现实世界能力生态系统之间的行动接口。*

这一定义捕捉了其本质角色：在代理决定做什么与这些决策如何通过能力执行实现之间进行中介。

## 7. 架构收益

### 7.1 解耦

应用程序变得独立于特定的提供者。代理表达其能力需求，而无需知道哪个提供者将满足这些需求。这种解耦实现了以下优势：

- 无需修改应用程序即可替换提供者
- 多供应商策略
- 提供者不可用时的优雅降级

**案例研究：提供者故障转移。** 考虑一个使用DALL-E的图像生成服务。当OpenAI遇到中断时，接触层可以将请求路由到Stable Diffusion（通过不同的提供者），而无需任何应用程序代码更改。执行层将相同的"文本到图像"能力请求翻译为替代提供者的API格式。这种故障转移发生在能力层面，而非网络层面。

### 7.2 可移植性

AI应用程序可以跨不同的运行时环境运行。只要运行时实现了AI协议，应用程序无需修改即可部署。这种可移植性类似于容器镜像可以在任何容器运行时上运行 [4][6]。

**跨语言验证。** ai-lib项目在四种编程语言上验证了这种可移植性：

- *ai-lib-rust：* 具有零成本抽象的系统编程，适用于高性能服务。
- *ai-lib-python：* 数据科学和ML工作流，与NumPy和PyTorch生态系统集成。
- *ai-lib-ts：* Node.js应用程序和通过WASM编译的浏览器端AI。
- *ai-lib-go：* 具有原生Kubernetes集成的云原生服务。

所有四个运行时共享相同的能力请求/响应语义，并通过通用的合规性测试套件验证。

### 7.3 能力发现

新的提供者可以注册能力，而无需修改应用程序。能力注册表作为动态发现机制，实现了以下功能：

- 新的能力提供者进入生态系统
- 应用程序在运行时发现新能力
- 无需集中协调的生态系统增长

**生态系统规模。** ai-protocol生态系统目前支持30+ AI提供者，具有标准化的、配置驱动的接口。新提供者贡献符合v2模式的能力清单；运行时自动发现并验证这些清单，无需代码更改。

### 7.4 生态系统增长

标准化协议实现了第三方扩展和工具。最小运行时设计创造了清晰的扩展点：

- 自定义能力提供者
- 替代编排策略
- 专用运行时实现
- 监控和可观测性工具

### 7.5 运维收益

除了架构收益外，该设计还带来了实际的运维优势：

**调试清晰性。** 当能力调用失败时，边界清晰地指示了问题所在的位置：

- 协议级错误：清单验证、模式不匹配。
- 翻译级错误：特定提供者的格式问题。
- 提供者级错误：速率限制、认证、服务中断。

**测试隔离。** 三层分离实现了独立的测试：

- 认知层测试关注提示工程和代理逻辑。
- 策略层测试关注路由决策和回退链。
- 执行层测试关注协议合规性和提供者翻译。

这种隔离降低了测试复杂性，并实现了有针对性的回归测试。

## 8. 相关工作

### 8.1 AI编排框架

LangChain等框架提供了实用的编排抽象，但经常将编排策略与执行机制耦合在同一个框架运行时中。这种耦合提高了开发者的便利性，但削弱了边界清晰性和运行时的可移植性。

**具体耦合问题。** 在LangChain中，提示模板、模型选择和输出解析经常在链定义中交织：

- *特定模型的提示。* 模板经常嵌入特定提供者的参数（如OpenAI函数调用语法），使得提供者替换变得非平凡。
- *隐式路由。* LangGraph条件边将路由逻辑嵌入图定义中，将策略决策与执行流混为一谈。
- *紧工具集成。* 工具定义经常引用特定模型能力（如视觉、函数调用），降低了在不具备这些功能的模型间的可移植性。

AI执行层明确分离了这些关注点：提示由认知层管理，路由由接触层处理，调用由执行层负责。

### 8.2 模型上下文协议（MCP）

Anthropic的模型上下文协议为AI模型与外部工具和资源交互提供了标准化接口。MCP与AI执行层解决了互补的关注点：

- *MCP范围。* MCP关注模型-工具接口，定义了模型如何通过JSON-RPC协议发现和调用工具。
- *执行层范围。* 本工作关注运行时-提供者接口，定义了执行层如何将能力请求翻译为特定提供者的API调用。
- *互补定位。* AI执行层可以将MCP作为其能力协议之一实现，将MCP工具调用翻译为特定提供者的实现。

MCP标准化了"模型请求，工具响应"的交互模式；AI执行层标准化了"运行时请求，提供者响应"的调用模式。

### 8.3 OpenAI Assistants和Agents SDK

OpenAI的Assistants API和Agents SDK为带工具使用的多轮对话提供了托管编排。关键架构差异如下：

- *状态管理。* Assistants API维护服务器端对话状态，包括消息历史和工具输出。AI执行层明确将状态管理排除在其边界之外。
- *提供者锁定。* Assistants API特定于OpenAI模型。执行层的协议优先设计实现了跨提供者的可移植性。
- *抽象级别。* Assistants API结合了认知（线程管理）、策略（工具选择）和执行（API调用）。三层分离实现了每层的独立演进。

### 8.4 工作流引擎

Airflow和Temporal等工作流引擎提供了强大的DAG执行和故障处理功能。然而，它们的抽象单位通常是任务/工作流级的编排，而非具有提供者无关协议契约的能力级语义调用。

**Temporal比较。** Temporal的工作流编排处理具有持久执行的长时间运行进程。然而：

- Temporal工作流在应用逻辑层面定义，而非能力调用层面。
- Activity实现通常是单语言的，而AI执行层需要跨语言的一致性。
- Temporal不解决提供者无关的能力翻译问题。

### 8.5 服务网格和RPC系统

服务网格和RPC基础设施在网络请求粒度上提供了路由、重试和可观测性。相比之下，AI执行层在能力语义层面运作，并明确分离了策略路由与确定性调用。

**语义与网络抽象。**

- *服务网格。* 基于服务身份和网络级策略路由HTTP/gRPC请求。不了解能力语义。
- *AI执行层。* 基于语义声明（如"文本到图像生成"）路由能力请求。在协议标准请求和特定提供者API之间进行翻译。

### 8.6 无服务器和函数运行时

FaaS平台提供了可扩展的执行环境，但不标准化能力抽象、能力清单或跨提供者的语义等价。

### 8.7 本文定位

本文为AI原生系统贡献了一个最小执行边界：

- 协议优先的能力调用
- 严格的认知-策略-执行分离
- 跨提供者和语言的运行时级可移植性
- 明确的非职责，防止策略泄漏到执行中

### 8.8 多语言实现

最小运行时设计的关键验证是其跨多种编程语言的可实现性。参考实现位于 https://github.com/ailib-official：

- **Rust (ai-lib-rust)：** https://github.com/ailib-official/ai-lib-rust — 规范实现，展示了类型安全的能力处理、零成本抽象和用于协议翻译的 `CapabilityDescription` 模式
- **Python (ai-lib-python)：** https://github.com/ailib-official/ai-lib-python — AI/ML工作流的主要实现，与流行的AI框架原生集成
- **TypeScript (ai-lib-ts)：** https://github.com/ailib-official/ai-lib-ts — Web和Node.js环境的实现，支持浏览器端和服务器端AI应用程序
- **Go (ai-lib-go)：** https://github.com/ailib-official/ai-lib-go — 云原生和微服务架构的实现

性能基准测试位于 https://github.com/ailib-official/ai-lib-benchmark，提供了跨不同运行时实现和提供者配置的延迟和吞吐量测量。

**基准测试结果：**

*表1：跨运行时性能（DeepSeek API）[^1]*

| 实现 | 吞吐量 (RPS) | 平均延迟 (ms) | P99 延迟 (ms) |
|------|-------------|--------------|---------------|
| ai-lib-rust | 99.1 ± 0.4 | 50.0 ± 0.2 | 201 ± 3 |
| ai-lib-python | 98.0 ± 0.5 | 51.0 ± 0.2 | 210 ± 4 |
| ai-lib-ts | 99.7 ± 0.2 | 49.6 ± 0.1 | 199 ± 2 |

*表2：提供者可移植性验证（Groq API）[^2]*

| 实现 | 吞吐量 (RPS) | P50 延迟 (ms) | P99 延迟 (ms) |
|------|-------------|--------------|---------------|
| ai-lib-rust | 17.9 ± 1.5 | 195 ± 2 | 1518 ± 274 |
| ai-lib-python | 20.4 ± 2.6 | 186 ± 1 | 1038 ± 348 |
| ai-lib-ts | 18.0 ± 0.9 | 195 ± 3 | 1611 ± 59 |

[^1]: 测试配置：DeepSeek API (`deepseek-chat`)，10秒时长 × 3次运行，5个并发连接，generate_text能力，100 max tokens。测试于2026-02-25。
[^2]: 测试配置：Groq API (`llama-3.1-8b-instant`)，10秒时长 × 3次运行，5个并发连接，generate_text能力。测试于2026-03-13。

**关键发现：**

1. **跨运行时一致性：** 所有三个实现在针对同一提供者时达到了统计上等效的吞吐量（2%以内方差），验证了协议级的一致性。
2. **提供者可移植性：** 相同的代码库成功连接到多个LLM提供者（DeepSeek、Groq）而无需修改，在实践中展示了能力抽象模型。
3. **生产可行性：** 吞吐量在DeepSeek上超过99 RPS，在Groq上超过17 RPS，尾部延迟适合交互式应用程序。

完整的基准测试配置、脚本和回归阈值可在基准测试仓库中找到。

跨语言一致性通过以下方式保持：

- 共享协议规范（YAML/JSON清单）
- 语言无关的能力模式
- 标准化请求/响应格式
- 验证行为等价性的通用测试套件

这种多语言支持展示了最小运行时抽象真正是语言无关的，适合生态系统范围的采用。

## 9. 实现考虑

### 9.1 运行时接口

执行运行时暴露了最小接口：

```python
interface ExecutionRuntime {
    # Execute a capability
    Result execute(CapabilityRequest request)

    # Query capability availability
    List<Capability> listCapabilities()

    # Health check
    HealthStatus healthCheck()
}
```

### 9.2 能力请求格式

能力请求遵循标准化的格式：

```json
{
    "capability": "generate_text",
    "version": "1.0",
    "input": {
        "prompt": "Explain quantum computing",
        "max_tokens": 500
    },
    "context": {
        "session_id": "abc123",
        "request_id": "req456"
    },
    "provider": {
        "name": "provider_a",
        "endpoint": "https://api.provider-a.com"
    }
}
```

### 9.3 结果格式

结果被标准化为带有执行指标的标准结构：

```json
{
    "capability": "generate_text",
    "status": "success",
    "output": {
        "text": "Quantum computing is...",
        "tokens_used": 450
    },
    "metadata": {
        "provider": "provider_a",
        "latency_ms": 1250,
        "translation_latency_ms": 15,
        "retry_count": 0,
        "timestamp": "2026-03-15T10:30:00Z"
    }
}
```

`retry_count`和`translation_latency_ms`字段提供了对执行行为的可见性，使接触层能够做出数据驱动的路由决策。

## 11. 局限性与评估议程

本工作关注架构形式化和运行时边界。除了评估议程，我们承认几个设计权衡和已知局限。

### 11.1 设计权衡

**最小与功能完整。** AI 执行层有意将其职责限制为确定性调用。这种最小性实现可移植性和清晰边界，但牺牲了集成框架中的便利功能：

- *无内置策略逻辑。* 路由决策、回退链和成本优化必须由接触层实现，而非运行时本身。
- *无对话管理。* 聊天历史、上下文窗口和记忆在执行边界之外。
- *无内置可观测性栈。* 提供指标、追踪和日志钩子，但可观测性基础设施是部署特定的。

**协议与实现。** 设计优先考虑协议稳定性而非实现便利性。能力清单必须显式声明；没有"通过内省自动发现"机制。这种权衡减少了运行时魔法，但增加了提供者接入工作量。

### 11.2 已知边界

该架构不适用于所有场景：

- *单一提供者锁定。* 使用标准化能力之外的特定提供者功能的应用程序失去可移植性收益。
- *超低延迟要求。* 协议翻译层增加的开销不适合亚毫秒响应场景（如高频交易）。
- *纯离线环境。* 虽然支持边缘运行时，但能力发现和提供者注册假设初始化期间有网络连接。

### 11.3 依赖假设

该架构依赖几个假设：

- *协议标准化。* 如果提供者不采用通用能力模式，收益会减少。
- *提供者合作。* 能力清单必须准确描述提供者行为；不匹配的清单导致运行时故障。
- *编排成熟度。* 没有接触层专业知识的组织可能无法完全实现分离收益。

### 11.4 评估议程

完整的系统评估应包括：

1. **翻译开销：** 协议到提供者转换的延迟和方差。
2. **运行时可扩展性：** 混合同步/流式工作负载下的吞吐量和尾部延迟。
3. **故障行为：** 提供者不稳定时有界重试的影响和错误传播。
4. **策略-执行分离成本：** 动态路由下控制面/数据面交互开销。

这些测量留给后续实验工作和跨运行时基准测试套件。

---

## 12. 结论

AI 代理的出现需要一个专门的能力执行架构边界。本文介绍了 AI 执行层作为代理驱动系统的最小运行时架构。架构通过逻辑模型 `S = (A, P, E)` 分离认知、编排和执行。

在该边界内，运行时被限制为六个执行职责：

1. 能力调用
2. 协议翻译
3. 结果标准化
4. 连接处理
5. 错误传播
6. 会话传输

理论基础由四个原则定义：

1. **能力抽象：** 代理依赖能力，而非提供者
2. **认知-行动分离：** 智能与执行分离
3. **能力路由：** 执行基于上下文动态路由
4. **能力图执行：** 复杂任务分解为能力图

执行层作为 AI 认知与现实世界能力生态系统之间的行动接口。该边界将代理与提供者解耦，提高应用程序可移植性，并支持生态系统规模增长。

更广泛地说，提出的层类似于早期计算时代的操作系统和 RPC 边界：它跨异构提供者提供稳定的执行契约，同时保留更高层的策略自主性。随着生态系统的成熟，该边界有望成为 AI 原生软件基础设施的基础组件。

## 附录 C：运行时伪代码

### C.1 执行能力

```python
def execute_capability(request: CapabilityRequest) -> Result:
    """
    Execute a capability request.

    Args:
        request: Standardized capability request

    Returns:
        Normalized execution result
    """
    # 1. Validate request
    validate_request(request)

    # 2. Translate to provider-specific format
    provider_request = translate_request(request)

    # 3. Invoke provider API
    try:
        provider_response = invoke_provider(
            endpoint=request.provider.endpoint,
            data=provider_request
        )
    except ConnectionError as e:
        # Propagate error (no retry logic)
        raise ExecutionError(
            capability=request.capability,
            error_type="connection",
            message=str(e)
        )

    # 4. Normalize response
    result = normalize_response(
        capability=request.capability,
        provider_response=provider_response
    )

    # 5. Return standardized result
    return result
```

### C.2 请求翻译

```python
def translate_request(request: CapabilityRequest) -> dict:
    """
    Translate standardized request to provider-specific format.

    Args:
        request: Standardized capability request

    Returns:
        Provider-specific request payload
    """
    capability_manifest = load_manifest(request.capability)
    provider_adapter = get_adapter(request.provider.name)

    return provider_adapter.translate(
        capability=request.capability,
        input_data=request.input,
        manifest=capability_manifest
    )
```
